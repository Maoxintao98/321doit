# 321Doit 安全性与完整性白皮书

**版本：** 0.6 · 2026-06-30
**范围：** 本文只描述 321Doit 影视制作全能工作站中“极速拷卡”模块的安全边界，不代表分镜、统筹、场记或媒体转换模块的完整产品说明。

**读者：** 现场 DIT、数据管理员、后期工程师、归档人员——任何需要确切了解 321Doit 拷贝行为提供了什么保证、又没有保证什么的人。

本文档描述 321Doit 的威胁模型、它实际给出的完整性保证、以及——明确地——它**不**保护什么。我们宁可范围窄一点说清楚，也不愿意大而化之。

英文版见 [SECURITY.md](SECURITY.md)。

## 1. 威胁模型

321Doit 的“极速拷卡”模块防御以下情况：

- **存储介质静默位翻转**：摄影机卡或目标盘在拷贝过程中悄悄返回了被破坏的字节。
- **写入中断**：断电、强行拔卡、进程崩溃导致的半成品文件被误认为完整文件。
- **意外覆盖**：后一张卡的下盘包覆盖了之前同名文件夹的下盘包。
- **操作员遗漏**：某个目标盘中途失败但没有被人注意到。
- **校验缺口**：报告说"拷贝成功"但事实上根本没有从目标盘读回来比对。

321Doit **不**防御以下情况：

- **有写入权限的恶意攻击者主动篡改**：xxHash64 不是密码学哈希，对手如果手里有清单，就可以构造出与原文件同 hash 的不同文件。SHA-256 模式（偏好里可选）能在数据层面防御这一点，但清单本身没有可信权威的签名。
- **被植入后门的系统组件**（ffmpeg、操作系统的文件 API、磁盘固件）。321Doit 假定本地环境是可信的。
- **下盘包被偷走或被擅自复制**。下盘包不做任何静态加密。
- **法律级别的不可抵赖性**。PDF 报告里有签名行，可以用墨水签字或第三方数字签名工具盖章，但 321Doit 自己**不**对 PDF 做密码学签名。

这条边界是有意收紧的。片场工作流追求的是速度和简洁，而不是密码学强度。需要更强保证（法律证据、十年级归档完整性）的场景，应当把 321Doit 的输出再叠加一层独立工具——例如官方 ASC MHL v2.0 的 C4 链路 hash（321Doit 已经写出，并且能被官方工具直接验证），再加一份独立的对清单的 detached 签名。

## 2. Hash 算法

| 算法      | 是否默认 | M 系列芯片吞吐 | 防位翻转 | 防恶意碰撞     |
|-----------|----------|-----------------|----------|----------------|
| xxHash64  | 是       | GB / 秒级       | 是       | **否**（非密码学）|
| MD5       | 可选     | 数百 MB / 秒    | 是       | 弱（已破）     |
| SHA-1     | 可选     | 数百 MB / 秒    | 是       | 弱（已破）     |
| SHA-256   | 可选     | 数百 MB / 秒    | 是       | 是             |

**为什么 xxHash64 是默认值。** DIT 在片场最痛的一件事是：256 GB 的卡乘以 3 个目标，校验阶段就要从盘上读回 768 GB，外加原本的写入。SHA-256 会让这一步成为整个任务的瓶颈；xxHash64 把这一步压回到磁盘速度。Pomfort Silverstack、ShotPut Pro 等行业内出货工具默认值也是 xxHash64，这意味着 321Doit 写出的清单与这套生态可以互通。

**什么时候改用 SHA-256。** 当下盘包要喂给归档管线（LTO 磁带、冷存储、长期法律保留）时——降低碰撞风险足以抵消吞吐损失，请在偏好里改用 SHA-256。

**为什么不用 BLAKE3 或 xxh3-128。** 两个都是好算法，但都还不是 ASC MHL v2.0 标准里的一等公民，写出来下游工具解析不了。等标准接纳后再考虑。

## 3. 拷贝与校验流程

每个文件、每个目标，都走这五步：

1. **打开临时文件。** 在目标盘同目录下打开一个 `.321doit-copying-<uuid>-<原文件名>`。其他工具按文件名找不到它；上一次任务崩溃留下的孤儿临时文件，会在下一次任务开始时被清理。
2. **流式写入。** 来源文件读一次，并行扇出到所有目标。默认 1 MiB 缓冲，偏好里可调。
3. **同步算 hash。** 选定的算法在字节流过的时候就算。来源文件每个文件**只读一次**，与目标数量无关。
4. **回读校验。** 临时文件关闭后，321Doit 重新打开它，从盘上读回来再算一次 hash。这是为了识破"消费级 SSD 或老化卡谎报写入成功"这一真实存在的失败模式。
5. **原子改名。** 只有当 hash 匹配时，临时文件才 `rename(2)` 到最终名字。在这一步之前任何崩溃都只会留下临时文件——最终名字根本不会出现。

某个目标校验失败时，只有那个目标被标记失败，其他目标继续。失败目标的清单里会显式记录那次 hash 不一致。

## 4. 防覆盖策略

输出目录名按 `<PROJECT>_<YYYYMMDD>_<CARD>` 决定（剪辑交付模式追加后缀）。如果这个目录在目标盘上已经存在，321Doit 直接以 `OffloadError.duplicateOutput` 拒绝任务，**不**做任何合并或覆盖。

理由：两次不同卡号的下盘悄悄合并到一起后，事后是无法挽救的。让操作员要么换卡号、要么把旧目录挪开——这是有意制造的摩擦点。

检查按目标分别进行，所以一个目标盘上有冲突、其他没有，只有冲突的那个会失败。

## 5. ASC MHL v2.0 合规

321Doit 在 `<下盘包>/ascmhl/` 下写出符合 `urn:ASC:MHL:v2.0` 命名空间的 v2.0 generation。同一目录下的链路文件 `ascmhl_chain.xml` 包含每一份 generation .mhl 的 C4（Cinema Content Creation Cloud）ID hash——它是 SHA-512 的 base58 编码，使用 C4 字符表，左侧用 "1" 补齐到 88 字符。

这意味着第三方——归档单位、后期公司、下一位 DIT——可以用 ASC 官方的 Python 工具直接验证下盘包：

```sh
pip install ascmhl

# 解析 schema + 验证 chain hash，检测结构性篡改
ascmhl info  /Volumes/Backup/PROJECT_YYYYMMDD_CARD

# 检查文件清单是否完整（多/少文件）。1.2 版本的 diff
# 不会重新算文件内容的 hash —— 详见下文说明。
ascmhl diff  /Volumes/Backup/PROJECT_YYYYMMDD_CARD

# 内容级 bit-rot 检测：重新算每个文件的 hash，发现不一致时
# 在 stderr 输出 "ERROR: hash mismatch …" 并以非零状态退出。
ascmhl create -v -h xxh64 /Volumes/Backup/PROJECT_YYYYMMDD_CARD
```

ascmhl 1.2 里 docstring 提到的 `verify` 子命令尚未发布，**真正能做内容级校验的入口是 `ascmhl create -v`**——它会写一份新的 generation，并把每一个文件的 hash 跟上一代比对。321Doit 在发版 smoke test 里跑的是 `ascmhl info` + `ascmhl diff`：这能抓住所有 schema 回归和文件清单出入，但**不会**抓到下盘包**已经写出后**的位翻转。需要在归档前再做一次内容级抽检的话，请按上面那条 `ascmhl create -v` 手动跑一次。

321Doit **内部**的下盘流程**确实会校验文件内容**：每个目标盘在 rename 之前会被回读、重算 hash 与来源比对（见上方"拷贝与校验流程"）。官方工具是对**清单**的第三方核对，不是对**拷贝**步骤的核对。

清单里的 ignore 列表覆盖了 321Doit 自己的报告与流程附件（`03_REPORTS/`、`04_CHECKSUMS/`、`_321Doit/`、转码相关目录），所以事后往下盘包里加一份 PDF 报告，不会让清单失效。

## 6. 报告

每次下盘都会在 MHL 旁边写一份 PDF 报告。报告包含：

- 标题页与每一页页脚的 "由 321Doit v0.x 生成" 戳记。
- **清单完整性 / Manifest Integrity** 板块，列出 .mhl 的 SHA-256。下游审核员可以据此确认拿到的清单确实是当时写出来的、没有被改过。
- **操作员签收 / Operator Sign-off** 板块：操作员姓名 + 一行签名/日期空白行。意图是在打印件上手写签字，或用 Preview / Adobe Acrobat 等工具盖章。321Doit 自己**不**给 PDF 嵌入 PKCS#7 签名——那需要分发或生成 X.509 证书，目前不在范围内。

## 7. 代码签名现状

321Doit 出货时是 **ad-hoc 代码签名**。我们没有为 Apple Developer ID 付费，所以 .app 和 .dmg 不带 Team ID，也没有 Apple notarization。macOS Gatekeeper 在新机器上第一次打开时会弹警告。README 已经说明右键 → 打开的步骤，必要时使用 `xattr -dr com.apple.quarantine`。

对用户的影响：

- **打开 .dmg 之前请用 GitHub Release 页面公布的 SHA-256 自行验证** dmg 文件。ad-hoc 签名意味着没有任何回到 Apple 的信任链。
- **自动更新（Sparkle）依赖独立于 Apple 体系的 EdDSA 签名**，所以更新链路是受保护的——尽管首次安装链路并不是。
- 未来如果买了 Developer ID，同一份二进制会以 notarized 签名重新发布，MHL 与 PDF 的输出不会有任何变化。

## 8. 已知局限

- **无失败卡的断点续传。** 失败的卡需要整张重跑。`_321Doit/session.json` 里有部分状态，引擎据此跳过已经校验通过的文件，所以重跑不是完整重拷。
- **R3D / BRAW / ARRIRAW / CRM 转码** 依赖本机 ffmpeg 的解码能力。321Doit 不附带 ffmpeg。
- **不在 App Store sandbox 里。** App 在用户权限范围内可读可写。卸载只删 .app bundle，不会动之前的下盘包。
- **静态不加密。** 下盘包以明文形式留在目标盘上。
