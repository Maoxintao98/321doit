# 321Doit 0.7 全仓成熟度审计

## 审计目标

0.7 不增加新的业务工具或宣传性功能。本轮审计只补齐现有五项工具作为日常工作软件所需要的可靠性、可恢复性、可诊断性、发布一致性和诚实的产品边界。

审计范围包括 macOS 主程序、iPad 场记伴随端、安装脚本、更新器、发布脚本、ASC MHL 与 Resolve 交叉验证、README、应用内文案和现有自动化测试。产品事实以 `PRODUCT.md` 与 `ToolRegistry` 为准。

## 本轮已经补齐

### 安装、更新与回滚入口

- 正式产物统一为 `DMG → 安装 321Doit.pkg → /Applications/321Doit.app`，不再把“拖入 Applications”写成当前正式安装方式。
- PKG `postinstall` 无论电脑是否已有 FFmpeg，都会先检查 App、版本、Build、Universal 2 架构和 ad-hoc 签名，再清除已安装 App 的 quarantine；修复了“检测到外部 FFmpeg 后提前退出，跳过隔离处理”的错误。
- 安装结果与失败原因写入 `/Library/Application Support/321Doit/install.log`，当前状态写入同目录 `install-status.txt`。
- 更新器支持同一短版本号下更高 Build，先下载到唯一缓存，再对同一份文件做 Ed25519 校验；只有验证通过的精确 DMG 才会被清除 quarantine 并打开安装。
- 已验证更新缓存可以从“工具”菜单安全清理；该操作不触碰项目、素材、报告、日志和未完成任务。
- appcast 保留历史 release item，继续提供旧版本下载入口。正式构建会生成候选 appcast，而不是直接改动公开更新源。

### 发布流水线

- `package.sh` 默认先运行严格测试，再构建 Universal 2 App、Universal 2 FFmpeg/FFprobe、PKG 和 DMG。
- 发布预检会阻断版本、Build、架构、签名策略、Hardened Runtime、公开密钥、DMG 名称、文件长度、SHA-256、Ed25519 签名、下载文件名或 appcast 不一致的候选包。
- 普通内部构建每次成功后自动递增 Build，失败不占号；正式复现可用 `APP_BUILD_OVERRIDE` 锁定。内部安装拒绝同一版本下相同或更低 Build 覆盖。
- 当前发布策略被明确锁定为 ad-hoc；脚本若发现 Developer ID 签名或 Hardened Runtime 状态与当前策略不一致会失败，避免生成“半签名”产物。
- Sparkle Ed25519 签名工具已适配当前 Swift/SDK 组合，并能重新验证已发布安装包。

### 数据安全与恢复

- 项目主快照和故事板保存前会保留经过解码验证的上一版本备份，默认最多保留 20 份，并用时间窗口避免高频写入放大。
- 主项目或故事板 JSON 损坏时会自动寻找最近一份有效备份；不支持的新 schema 会硬失败，禁止把新数据误当成旧格式覆盖。
- 设置文件损坏时会回退 `settings.backup.json`；损坏设置最多保留 3 份用于诊断。主文件和备份均失效时使用安全默认值，并在“日志与诊断”显示警告。
- 退出应用前主动刷新延迟中的设置和场记项目保存。
- 媒体转换队列、最近项目索引损坏时不再静默失败：无效索引会被隔离/丢弃并写入应用日志，避免每次启动重复失败。

### 日志与支持

- 应用日志默认位于 `~/Library/Application Support/321Doit/Logs`，默认保留 30 天；用户可在设置中修改位置、保留天数和详细程度。
- 生命周期、更新、拷卡、媒体转换、设置恢复、项目恢复、Webhook、系统通知和后期自动交接失败会进入统一日志。
- “联系与支持”可导出诊断 ZIP。诊断包包含版本、系统、脱敏设置摘要、最近应用日志和可读取的安装日志；不会复制素材、项目内容、报告、待恢复任务、安全作用域书签或钥匙串 Webhook 密钥。日志可能包含本机路径，压缩包内附隐私提示。

### 产品边界与基本可访问性

- 删除云同步、团队协作、“更多工具即将加入”和未配置 OpenCode 的禁用入口；不把接口缝隙展示成已交付或即将交付的能力。
- 关于页、工作台品牌栏、README 中英文和安装说明统一到五项工具、本地优先的当前产品事实。
- 工具卡、工作区切换、项目顶栏和支持按钮恢复键盘焦点，保留现有快捷键工作流。

## 当前明确不具备

以下内容不是遗漏，而是继续使用 ad-hoc 发布策略的直接结果：

- 没有 Apple Developer ID Application / Installer 签名。
- 当前 App 与 DMG 使用 ad-hoc 签名；PKG 没有可用的 ad-hoc Installer 签名类型，因此保持 unsigned，发布脚本会校验这个状态。
- 没有 Apple Notarization 公证。
- Hardened Runtime 保持关闭。
- 浏览器、邮件或聊天工具手动下载的 DMG/PKG 仍可能被 Gatekeeper 拦截；PKG 只改善安装结构，不能替代 Developer ID 与公证。
- 应用内更新路径可以在加密校验后只清除同一个已验证 DMG 的隔离属性，但不能把这种项目自校验描述成 Apple 公证。

这些限制必须继续出现在 README、发布说明和测试说明中。取得 Developer ID 后，应在独立版本中同时完成 App、PKG、DMG 签名策略迁移、Hardened Runtime、notarytool 提交、staple 和干净机器 Gatekeeper 验收，不能只打开其中一项。

## 仍需真实环境验收

代码和自动化无法替代以下人工证据，因此这些项目是 0.7 RC 门槛，不应伪造为已完成：

- Intel Mac 与 Apple Silicon Mac 的全新用户冷安装、覆盖安装、同版本更高 Build 安装和旧版本回滚安装。
- Safari/Chrome/邮件/网盘四类下载来源的 quarantine 与 Gatekeeper 行为。
- APFS、HFS+、exFAT，以及不同容量相机卡/读卡器上的断电、拔盘、空间不足、只读、休眠、取消和严格续传。
- 五项工具分别完成新建、保存、关闭、恢复、导入和导出；中英文界面完成键盘和 VoiceOver 冒烟。
- 用真实 Resolve、Final Cut Pro、ASC MHL 接收方和真实相机素材完成交接验证。

## 1.0 之前的判断

从代码和发布工程看，0.7 已经进入“可靠性收口”阶段，而不是继续堆工具。距离 1.0 的主要工作量不在新增功能，而在三轮证据积累：故障注入、真实设备矩阵和 RC 冻结。

如果能稳定获得 Intel/Apple Silicon 测试机、三种文件系统存储和真实片场样本，每轮问题都只做修复不扩范围，预计还需要约 **6–10 周**：

1. 2–3 周完成故障恢复矩阵；
2. 2–4 周完成真实设备、安装来源和后期软件矩阵；
3. 2–3 周完成 RC 冻结、文档、回滚演练与发布候选验证。

若继续 ad-hoc，1.0 可以作为“功能与数据稳定的正式版”发布，但安装体验仍会有明确限制。若把“普通用户双击安装、无 Gatekeeper 绕行”也定义为 1.0 门槛，则时间还取决于 Developer ID 申请与 Apple 公证链路，不能由代码工期单独承诺。
